# 《Risk Register v0.1》

本文件记录 First Playable 的主要风险和降级策略。

---

## 1. 范围风险

| 风险 | 等级 | 表现 | 降级策略 |
|---|---|---|---|
| 内容扩张过快 | 高 | 想做第二大阶段、更多法宝、完整魂修 | 严格使用 Feature Scope Matrix，非 Must 延后 |
| 局外系统过大 | 高 | 弟子、宗门、市场、交易抢先进入 | 局外只做洞府最小闭环 |
| 在线联机过早产品化 | 中高 | 先做 matchmaking/重连/公网服务 | v0.1 只要求 local + harness，Relay 可选 |

---

## 2. 设计风险

| 风险 | 等级 | 表现 | 降级策略 |
|---|---|---|---|
| 灵气经验和修为混淆 | 高 | 玩家以为顿悟等于境界突破 | UI 颜色/文案/触发结果强分离；测试中追问玩家 |
| 丹药被当血瓶 | 中 | 玩家残血才吃，感觉没用 | 降低瞬时反馈，强调炼化预警和 Boss 前提示 |
| 雷劫像随机秒杀 | 高 | 玩家看不清预警或伤害过高 | 增加预警、降低半径、调低伤害、减少叠加密度 |
| 法术无脑乱按 | 中 | 真元过多，主动性消失 | 提高消耗或降低真元掉落 |
| 玩家不放法术 | 中 | 真元太少或 UI 不清 | 降低 cost，提升真元球，增强 Ready 动效 |

---

## 3. 工程风险

| 风险 | 等级 | 表现 | 降级策略 |
|---|---|---|---|
| Simulation 非确定性 | 高 | 同 seed hash mismatch | 禁止 Math.random/Date.now/DOM；固定 tick；hash test |
| UI 直接读 SimState | 中 | 后续联机/回放难维护 | 必须经过 ViewStateBuilder |
| VFX 影响 gameplay | 高 | 粒子 RNG 改变掉落/敌人 | visualRng 独立；VFX 不进 hash |
| Canvas 与逻辑耦合 | 中 | Renderer 修改实体状态 | Renderer 只读 ViewState/EffectEvent |
| Codex 一次改太多 | 高 | 架构漂移、测试缺失 | 每任务限定修改范围和验收 |

---

## 4. 可玩性风险

| 风险 | 等级 | 表现 | 降级策略 |
|---|---|---|---|
| 第一阶段太长 | 中 | 试玩疲劳 | 压缩波次或 Boss HP |
| Boss 太短 | 中 | 无法验证丹药/救援 | 增加 HP 或阶段免疫窗口 |
| 怪潮堆满屏 | 中 | 小怪 TTK 太长 | 降 HP / 提普攻 DPS / 降 spawn |
| 特效挡弹 | 高 | 玩家看不清敌弹 | ReadabilityGuard、层级降透明、敌弹白芯 |
| 局外回报不明显 | 高 | 打完不想再来一局 | 确保第二局开局有星级/法术/丹药变化 |

---

## 5. 降级优先级

如果时间不足，按以下顺序降级：

```text
1. 完整 WebSocket Relay → 降级为 two-client local harness。
2. 局外劫雷台完整战斗 → 降级为入口 + 数据 + 不可挑战提示。
3. 程序音效 → 降级为静音但保留 cue 数据。
4. 低配模式 → 降级为手动 particleScale。
5. 局内雷劫正式自然触发 → 降级为 Debug-only。
6. 第二套 Loadout 预设 → 保留一套稳健开荒。
```

不能降级的项目：

```text
双轨成长分离
第一大阶段完整战斗
局外结算强化闭环
数据驱动
Seeded RNG
Headless simulation
```
